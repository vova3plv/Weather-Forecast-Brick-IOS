# Документація з лінтингу (SwiftLint)

## Обраний інструмент
Для статичного аналізу коду в проєкті "Weather Forecast Brick" використовується **SwiftLint**. 
**Причини вибору:**
1. Це де-факто індустріальний стандарт для iOS-розробки мовою Swift.
2. Базується на офіційних гайдлайнах стилю від GitHub.
3. Має безшовну інтеграцію з Xcode через Build Phases.
4. Підтримує автоматичне виправлення (autocorrect) багатьох стилістичних помилок.

## Базові правила
Конфігурація знаходиться у файлі `.swiftlint.yml` в корені проєкту. Основні правила:
* **Ігнорування:** З перевірки виключені папки `Pods`, тести та директорії збірки.
* **disabled_rules:** Відключено суворе обмеження на довжину рядка (`line_length`) та дозволено `trailing_whitespace` для більшої гнучкості під час розробки.
* **opt_in_rules:** Включено правило `empty_count` (використання `.isEmpty` замість `.count == 0` для кращої продуктивності) та `force_unwrapping` (попередження про потенційно небезпечне використання `!`).
* **Обмеження розміру:** Встановлено ліміти на довжину тіла функції (warning при 50 рядках, error при 100).

## Інструкція з запуску
1. Встановіть SwiftLint через Homebrew: `brew install swiftlint`.
2. Лінтер автоматично запускається під час кожної збірки проєкту в Xcode (інтегровано як Run Script у Build Phases). Всі попередження відображаються безпосередньо в редакторі коду.
3. Для ручного запуску з терміналу виконайте команду `swiftlint` у корені проєкту.
4. Для автоматичного виправлення безпечних стилістичних помилок виконайте `swiftlint autocorrect`.

## Git Hooks та Інтеграція зі збіркою
* **Інтеграція зі збіркою (Build Phase):** SwiftLint інтегровано безпосередньо в процес збірки Xcode за допомогою Run Script Phase. Це гарантує, що розробник бачить порушення стилю як попередження (або помилки) прямо в редакторі коду під час компіляції.
* **Pre-commit Hook:** У репозиторії налаштовано клієнтський Git Hook (`.git/hooks/pre-commit`), який автоматично запускає `swiftlint --strict` перед кожним комітом. Якщо в коді є серйозні порушення стилю, процес створення коміту переривається.
* **Статична типізація (Static Type Checking):** Оскільки розробка ведеться мовою Swift, яка є сильно та статично типізованою мовою, статичний аналіз типів виконується нативним компілятором `swiftc` "з коробки" під час кожної збірки (Build). Додаткові інструменти (як-от mypy для Python) не потрібні.

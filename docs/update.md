# Процедура оновлення додатку (Release Management)

## 1. Підготовка до оновлення
* **Резервне копіювання:** Зробити коміт та push усіх фінальних змін у гілку `main`.
* **Версіонування:** Збільшити номер версії за принципом Semantic Versioning (наприклад, з 1.0.0 до 1.1.0 для нового функціоналу, або 1.0.1 для багфіксів). Збільшити Build number на 1.
* Зібрати Release Notes (опис змін) для користувачів.

## 2. Процес оновлення
На відміну від веб-серверів, iOS-додатки оновлюються через App Store Review:
1. Виконати процедуру архівування та завантаження (див. `deployment.md`).
2. В App Store Connect створити нову версію додатку.
3. Обрати завантажену збірку, додати Release Notes та скріншоти.
4. Відправити додаток на перевірку Apple (Submit for Review).

## 3. Процедура відкату (Rollback) у разі невдалого оновлення (+1 бал)
В екосистемі iOS **неможливо** примусово знизити версію додатку на пристроях користувачів. Процедура "відкату" виглядає так:
1. Якщо критична помилка знайдена на етапі Phased Release (поетапного розгортання), негайно зупинити реліз в App Store Connect.
2. Створити гілку `hotfix` від останнього стабільного коміту.
3. Виправити критичну помилку.
4. Збільшити номер версії (наприклад, 1.1.1).
5. Зробити термінову збірку та подати її на розгляд Apple з позначкою "Expedited Review" (Пришвидшений розгляд для критичних виправлень).
